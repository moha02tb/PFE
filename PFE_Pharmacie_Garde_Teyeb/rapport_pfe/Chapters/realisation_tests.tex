\section{Introduction}

Après la phase de conception, ce chapitre présente le passage de PharmacieConnect vers une version
exécutable et testable. Il décrit l'environnement de développement, les choix techniques réellement
mobilisés, l'organisation du dépôt et les principales fonctionnalités implémentées dans le backend,
le tableau de bord administrateur, l'application mobile et l'assistant de premiers secours. L'objectif
n'est pas seulement d'énumérer les outils utilisés, mais de montrer comment chaque composant contribue
aux besoins identifiés : consultation des gardes, gestion administrative, import CSV, traçabilité,
géolocalisation et validation des accès.

\section{Réalisation backend}

Dans la réalisation, le backend est devenu le point de passage obligatoire entre les interfaces et la base de
données. Cette position lui permet de recevoir les requêtes web et mobiles, d'appliquer les règles métier et
de garder au même endroit l'authentification, la validation, l'audit et les imports.

\subsection{Endpoints publics}

Les endpoints publics répondent surtout aux besoins de l'application mobile. Ils donnent accès aux
informations de consultation sans passer par l'espace d'administration.

\begin{itemize}
    \item \texttt{GET /health} pour vérifier l'état du backend ;
    \item \texttt{GET /api/pharmacies} pour consulter les pharmacies avec pagination ;
    \item \texttt{GET /api/pharmacies/search} pour rechercher par texte ;
    \item \texttt{GET /api/pharmacies/nearby} pour rechercher les pharmacies proches d'une position GPS ;
    \item \texttt{GET /api/pharmacies/\{pharmacy\_id\}} pour consulter le détail d'une pharmacie ;
    \item \texttt{GET /api/gardes} pour consulter les gardes d'une date donnée ;
    \item \texttt{GET /api/medicines} pour rechercher dans le catalogue de médicaments ;
    \item \texttt{GET /api/medicines/\{code\_pct\}} pour consulter le détail d'un médicament.
\end{itemize}

Pour la recherche par proximité, l'API utilise les coordonnées GPS envoyées par le client. Elle calcule la
distance avec les pharmacies disponibles et retourne les résultats triés par distance.

\subsection{Authentification et sessions}

L'authentification est gérée côté backend afin de garder le contrôle sur les comptes et les autorisations.
Lorsqu'une connexion réussit, le serveur retourne un jeton d'accès et un jeton de rafraîchissement.

\begin{lstlisting}[caption={Structure simplifiée d'une réponse de connexion}]
{
  "access_token": "[jeton d'accès]",
  "refresh_token": "[jeton de rafraîchissement]",
  "token_type": "bearer",
  "expires_in": 900
}
\end{lstlisting}

Les routes protégées vérifient ensuite le jeton reçu et le rôle de l'utilisateur. Cette vérification est
réalisée côté serveur afin qu'un simple contrôle d'interface ne puisse pas être contourné.

\subsection{Services métier}

La logique principale est placée dans des services. Ce choix évite de concentrer trop de code dans les
routes et rend les traitements plus faciles à tester.

\begin{itemize}
    \item \texttt{AuthService} pour l'inscription, la connexion, la vérification email, le rafraîchissement et la déconnexion ;
    \item \texttt{PharmacyService} pour la gestion et l'import des pharmacies ;
    \item \texttt{GardeService} pour la gestion et l'import des plannings de garde ;
    \item \texttt{MedicineService} pour l'import, la recherche et la consultation des médicaments ;
    \item \texttt{AuditService} pour la journalisation des actions importantes ;
    \item \texttt{CacheService} pour l'utilisation du cache lorsque la configuration le permet.
\end{itemize}

\subsection{Imports CSV}

Les imports CSV ont été ajoutés pour éviter la saisie manuelle d'un grand nombre de lignes. Avant
l'enregistrement, le backend vérifie l'extension, la taille, les colonnes obligatoires, le format des
valeurs et les contraintes métier.

Lorsqu'une ligne est incorrecte, la réponse indique les informations nécessaires pour corriger le fichier.
Pour les médicaments, le code PCT sert d'identifiant pour mettre à jour un enregistrement existant au lieu
de créer un doublon.

\subsection{Audit et indicateurs}

Les actions sensibles sont enregistrées dans les journaux d'audit. Cela concerne notamment les
connexions, les échecs de connexion, les inscriptions, les créations administratives et les imports. Les
données statistiques alimentent le tableau de bord et donnent une vision synthétique de l'activité.

\section{Réalisation du tableau de bord administrateur}

Le tableau de bord administrateur rassemble les fonctions de gestion dans une interface web destinée aux
administrateurs et aux utilisateurs autorisés.

\subsection{Pages principales}

Les principales pages développées sont :

\begin{itemize}
    \item le tableau de bord pour les compteurs, indicateurs et activités récentes ;
    \item la page des pharmacies pour consulter, créer, modifier, supprimer et importer les pharmacies ;
    \item la page des gardes pour gérer les plannings de garde ;
    \item la page des médicaments pour importer et consulter le catalogue ;
    \item la carte pour visualiser les pharmacies géographiquement ;
    \item la page de management pour gérer les assistants et les autorisations ;
    \item la page des journaux d'audit pour consulter les actions enregistrées ;
    \item les pages de profil, paramètres, notifications et langues.
\end{itemize}

\subsection{Gestion des sessions côté administration}

Le tableau de bord utilise un client API centralisé. Ce client ajoute le jeton JWT aux requêtes protégées.
Si le jeton d'accès expire, il tente de renouveler la session. En cas d'échec, l'utilisateur est redirigé
vers la page de connexion.

\section{Réalisation application mobile}

L'application mobile correspond au parcours le plus visible pour le citoyen. Nous l'avons organisée autour
des usages les plus directs : rechercher une pharmacie, consulter les gardes, ouvrir la carte, parcourir le
catalogue de médicaments et ajuster quelques paramètres.

\subsection{Recherche et détail des pharmacies}

Depuis l'écran d'accueil, l'utilisateur peut rechercher une pharmacie par texte et consulter les résultats.
La fiche de détail affiche les informations disponibles, comme le nom, l'adresse, le téléphone, le
gouvernorat et les coordonnées lorsque celles-ci existent.

\subsection{Carte et géolocalisation}

L'application peut demander l'autorisation d'utiliser la position de l'utilisateur. Cette position est
envoyée au backend afin de récupérer les pharmacies proches. La carte affiche ensuite les résultats sous
forme de marqueurs.

\subsection{Calendrier des gardes}

Le calendrier donne à l'utilisateur le choix d'une date et affiche les pharmacies de garde correspondantes.
Cette fonctionnalité répond au besoin principal du projet : trouver rapidement une pharmacie disponible à
une date précise.

\subsection{Catalogue de médicaments}

L'écran des médicaments prend en charge la recherche par nom commercial, DCI ou code PCT. La fiche de détail
affiche les informations présentes dans la base, comme le prix public, le tarif de référence et la catégorie
de remboursement.

\subsection{Paramètres, langues et thème}

L'application prend en charge le français, l'anglais et l'arabe. Le choix de l'arabe active l'affichage RTL.
Les paramètres regroupent aussi le thème, les notifications, les favoris et certaines préférences
utilisateur.

\section{Réalisation de l'assistant conversationnel de premiers secours}

L'assistant conversationnel complète le parcours principal de PharmacieConnect en proposant une aide de
premiers secours à côté de la recherche de pharmacies de garde. Pour limiter le risque de réponses non
ancrées dans les données disponibles, l'implémentation privilégie une architecture de récupération
d'informations hybride : une recherche lexicale par BM25 \cite{robertson_okapi_trec3} et une recherche
sémantique fondée sur des embeddings \texttt{all-MiniLM-L6-v2} \cite{sentence_transformers_minilm}. Les
réponses sont donc construites à partir de passages récupérés, puis accompagnées d'un score de confiance
et d'avertissements pour les situations critiques.

\subsection{Architecture générale}

L'architecture de l'assistant suit un pipeline en plusieurs étapes :

\begin{enumerate}
    \item \textbf{Analyse de la requête} : Normalisation, tokenisation, détection de type et sujet.
    \item \textbf{Récupération parallèle} : Deux algorithmes complémentaires :
        \begin{itemize}
            \item \textbf{BM25 (Okapi)} : récupération lexicale sensible aux termes de la question ;
            \item \textbf{Embeddings sémantiques} : représentation vectorielle des questions et passages avec \texttt{all-MiniLM-L6-v2}.
        \end{itemize}
    \item \textbf{Fusion hybride} : Normalisation et fusion des scores :
        \begin{itemize}
            \item 0.55 pour BM25 (lexical) ;
            \item 0.35 pour embeddings sémantiques ;
            \item 0.10 pour bonus de correspondance thématique.
        \end{itemize}
    \item \textbf{Reclassement et extraction} : reclassement par similarité, alignement thématique et couverture, puis réponse extractive composée du texte récupéré.
    \item \textbf{Contrôle de confiance} : calcul multi-signaux (score, similarité, cohérence) et réponse de fallback lorsque la confiance est insuffisante.
    \item \textbf{Sécurité médicale} : Avertissement auto pour sujets à risque (étouffement, choc, hémorragie).
\end{enumerate}

\subsection{Données d'apprentissage et nettoyage}

Le jeu de données de départ s'appuie sur FirstAidQA, un corpus synthétique orienté premiers secours et
situations d'urgence \cite{firstaidqa}. Dans PharmacieConnect, ce corpus a été nettoyé, segmenté et
complété par des exemples ciblés afin d'obtenir un index mieux adapté aux questions courtes posées depuis
l'application mobile.

\begin{table}[H]
    \centering
    \caption{Statistiques du jeu de données de l'assistant -- Source : Auteur}
    \begin{tabular}{L{6.5cm}L{6.5cm}}
        \toprule
        \textbf{Métrique} & \textbf{Valeur} \\
        \midrule
        Questions/réponses brutes & 3,914 paires \\
        Documents indexés (chunks) & 5,270 segments \\
        Sources uniques & 3,898 documents \\
        Sujets (catégories) & 13 catégories (étouffement, saignements, brûlures, etc.) \\
        Ensemble d'entraînement nettoyé & 3,901 paires \\
        Ensemble de validation nettoyé & 409 paires \\
        Changements de labels (audit) & 1,841 sur l'entraînement, 165 sur la validation \\
        \bottomrule
    \end{tabular}
\end{table}

Le nettoyage des données a utilisé un audit automatisé (\texttt{data\_quality.py}) qui détecte et corrige les étiquettes erronées assignées d'après les termes de réponse, préserve les étiquettes originales dans \texttt{original\_topic} pour examen ultérieur, et génère un rapport d'audit détaillé avec traces de modification. L'augmentation ciblée de données a renforcé les sujets faibles (premiers secours généraux, saignements, morsures) en générant 60 exemples synthétiques pour améliorer la couverture et la robustesse.

\subsection{Entraînement et métriques de performance}

L'entraînement est piloté par le script \texttt{train\_firstaid\_chatbot.py}, qui construit le pipeline complet
et évalue les performances selon plusieurs métriques.

\begin{table}[H]
    \centering
    \caption{Performances de l'assistant après entraînement -- Source : Auteur}
    \begin{tabular}{L{3.5cm}L{2.2cm}L{2.0cm}L{4.3cm}}
        \toprule
        \textbf{Métrique} & \textbf{Valeur} & \textbf{Cible} & \textbf{Interprétation} \\
        \midrule
        Hit Rate / Recall@1 & 79.71\% & $\geq 75\%$ & Juste au rang 1 \\
        Recall@3 & 90.95\% & - & Juste dans top-3 \\
        Recall@5 & 94.38\% & $\geq 90\%$ & Juste dans top-5 \\
        MRR & 0.8627 & $\geq 0.80$ & Rang moyen inverse \\
        MAP@5 & 0.8572 & $\geq 0.78$ & Précision moyenne \\
        Evidence F1 & 0.2990 & $\geq 0.35$ & Extraction de span \\
        Fallback Rate & 2.69\% & $\leq 5\%$ & Refus confiance faible \\
        Smoke Tests & 14/14 & 100\% & Cas critiques \\
        \bottomrule
    \end{tabular}
\end{table}

\textbf{Analyse des résultats} :
\begin{itemize}
    \item la récupération documentaire est satisfaisante, avec un Recall@5 de 94.38\% ;
    \item le MRR de 0.8627 indique que les documents utiles apparaissent généralement dans les premiers rangs ;
    \item l'extraction précise de spans reste le point le plus faible, avec un Evidence F1 de 0.2990 ;
    \item l'écart observé vient surtout de l'appariement entre réponses attendues et passages récupérés ;
    \item le taux de fallback de 2.69\% montre que le mécanisme de confiance intervient dans les cas incertains.
\end{itemize}

La valeur d'Evidence F1 inférieure à la cible (0.2990 vs 0.35) s'explique principalement par le mode
d'appariement entre les réponses de référence et les documents indexés. Le pipeline extrait des passages
complets plutôt que des spans précis, ce qui pénalise le score F1 au niveau des tokens. Une amélioration
possible consiste à réétiqueter les données avec des spans d'extraction annotés manuellement, ou à
intégrer ultérieurement un module extractif supervisé, évalué sur des spans annotés, en remplacement de
l'extraction heuristique actuelle.

\subsection{API et intégration avec l'application mobile}

L'assistant est exposé via une API FastAPI dédiée, lancée sur un port séparé (8001 par défaut) pour éviter
le conflit avec l'API principale de pharmacies (8000).

\begin{lstlisting}[caption={Exemple de requête à l'API de l'assistant}]
POST /answer
Content-Type: application/json

{
  "query": "Comment faire face à l'étouffement ?",
  "top_k": 5
}
\end{lstlisting}

\begin{lstlisting}[caption={Structure de la réponse de l'assistant}]
{
  "query": "Comment faire face à l'étouffement ?",
  "answer": "Faites-le tousser vigoureusement.",
  "confidence": 0.92,
  "evidence_sources": [docid_1, docid_2],
  "safety_notice": "APPEL D'URGENCE REQUIS",
  "topic": "choking"
}
\end{lstlisting}

L'intégration côté mobile ajoute un onglet ou un écran dédié à l'assistant, permettant aux utilisateurs de
poser des questions de premiers secours tout en accédant à localisateurs de pharmacies de garde. Les
requêtes utilisateur sont envoyées à l'API de l'assistant, qui retourne la réponse enrichie avec confiance,
sources et avertissements de sécurité.

\subsection{Tests et validation}

L'assistant bénéficie de deux niveaux de validation : des smoke tests couvrant 14 scénarios d'urgence
courants, dont l'étouffement, le choc et la perte de conscience, ainsi que trois suites unitaires portant
sur la normalisation, la fusion hybride et le calcul de confiance. Comme les réponses sont extraites de
passages récupérés, le risque de contenu non ancré est réduit ; il n'est cependant pas présenté comme nul,
ce qui justifie la présence d'un score de confiance et de messages de sécurité.

\section{Déploiement et configuration}

La configuration du backend a été externalisée dans des variables d'environnement. Dans notre cas, cela évite
de mélanger le code applicatif avec des valeurs qui changent selon le contexte d'exécution, comme la base de
données, les origines autorisées ou les clés de sécurité.

\begin{table}[H]
    \centering
    \caption{Variables d'environnement principales -- Source : Auteur}
    \begin{tabular}{L{6.2cm}L{6.8cm}}
        \toprule
        \textbf{Variable} & \textbf{Rôle} \\
        \midrule
        \texttt{DATABASE\_URL} & Chaîne de connexion à la base de données. \\
        \texttt{SECRET\_KEY} & Clé utilisée pour signer les jetons JWT. \\
        \texttt{ALGORITHM} & Algorithme de signature des jetons. \\
        \texttt{ACCESS\_TOKEN\_EXPIRE\_MINUTES} & Durée de validité du jeton d'accès. \\
        \texttt{REFRESH\_TOKEN\_EXPIRE\_DAYS} & Durée de validité du jeton de rafraîchissement. \\
        \texttt{FRONTEND\_URL} & Origine autorisée pour l'interface web. \\
        \texttt{TRUSTED\_HOSTS} & Liste des hôtes autorisés. \\
        \texttt{ENABLE\_REDIS\_CACHE} & Activation ou désactivation du cache Redis. \\
        \bottomrule
    \end{tabular}
\end{table}

Le déploiement en production n'a pas été réalisé dans le cadre de ce PFE en raison des contraintes de
temps et de la nature du stage externe. L'architecture retenue est néanmoins compatible avec un
déploiement sur des plateformes cloud comme Railway ou Render pour le backend FastAPI, et Vercel pour
l'interface d'administration React/Vite. Ces étapes constituent une perspective concrète à court terme.

Pour la version de validation, le projet est exécuté localement de manière contrôlée : le backend FastAPI
est lancé avec Uvicorn, l'interface d'administration avec le serveur de développement Vite et
l'application mobile avec Expo. Dans cette configuration, l'intégration entre les trois parties peut être
vérifiée avant une mise en production. Pour un déploiement final, la même architecture peut être placée
derrière un serveur HTTPS, avec PostgreSQL comme base de données persistante et des variables
d'environnement séparées pour la production. Les commandes de lancement local sont placées en annexe afin
de ne pas alourdir le corps du rapport.

\subsection{Tests automatisés du backend}

Les tests backend sont écrits avec Pytest. Ils utilisent une base de test afin de ne pas modifier les
données de développement. Les fichiers identifiés couvrent notamment :

\begin{itemize}
    \item \texttt{test\_analytics\_dashboard.py} pour les indicateurs du tableau de bord ;
    \item \texttt{test\_assistant\_regions.py} pour les restrictions régionales des assistants ;
    \item \texttt{test\_audit\_logs.py} pour la traçabilité des connexions et des actions ;
    \item \texttt{test\_medicine\_upload.py} pour l'import et la recherche de médicaments.
\end{itemize}

Les tests d'audit vérifient qu'une connexion administrateur réussie crée une entrée
\texttt{ADMIN\_LOGIN}, qu'un échec crée une entrée \texttt{ADMIN\_LOGIN\_FAILED}, et qu'une inscription crée
une entrée \texttt{USER\_REGISTERED}. Les tests de restrictions régionales contrôlent qu'un assistant ne
peut agir que dans son périmètre autorisé. Les tests d'import de médicaments couvrent les colonnes
obligatoires, la normalisation des prix, la gestion des doublons par \texttt{code\_pct} et les messages
d'erreur pour les lignes invalides.

limites de pagination, les erreurs de validation, les accès interdits et la cohérence des réponses JSON.

\section{Conclusion}

La réalisation a abouti à une plateforme fonctionnelle composée d'une API sécurisée, d'un tableau de bord
administrateur et d'une application mobile. Les tests réalisés ont couvert les parcours essentiels :
authentification, recherche de pharmacies, consultation des gardes, imports CSV, gestion des rôles, audit
et recherche de médicaments. Des améliorations restent possibles, notamment les tests bout en bout, les
tests de charge et la validation sur un volume de données plus important.
